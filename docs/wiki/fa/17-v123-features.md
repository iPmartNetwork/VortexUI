# 17. VortexUI نسخه 1.2.3 — راهنمای ویژگی‌های جدید

!!! info "نسخه 1.2.3"
    این صفحه تمام ویژگی‌های معرفی‌شده در VortexUI نسخه 1.2.3 را مستند می‌کند. هر بخش توضیح می‌دهد که ویژگی چه کاری انجام می‌دهد، در کجای رابط کاربری قرار دارد و چگونه پیکربندی می‌شود.

---

## میزبان‌های اشتراک (Subscription Hosts)

**مکان:** شبکه و نودها → میزبان‌های اشتراک (از تب **Hosts** هر inbound نیز قابل دسترسی است)

بازنویسی میزبان به‌ازای هر inbound، به سبک Marzban. یک میزبان اشتراک آنچه را که لینکِ رندرشده در اشتراک کاربر تبلیغ می‌کند تغییر می‌دهد — آدرس، SNI، هدر Host، مسیر، امنیت ترنسپورت و موارد دیگر — **بدون** دست‌زدن به پیکربندی زنده‌ی هسته روی نود. این امکان می‌دهد یک inbound واحد را پشت چندین دامنه‌ی CDN، چند SNI یا چند پورت قرار دهید.

### آنچه می‌توانید بازنویسی کنید

| Field | توضیحات |
|-------|----------|
| Remark | نام نمایشی ورودی (از متغیرهای قالب پشتیبانی می‌کند) |
| Address | میزبان/IP تبلیغ‌شده به کلاینت‌ها (مثلاً یک دامنه‌ی CDN) |
| Port | بازنویسی پورت (`0` = ارث‌بری پورت inbound) |
| SNI | نام سرور TLS |
| Host | بازنویسی هدر `Host` |
| Path | مسیر WS/HTTPUpgrade/gRPC |
| ALPN | فهرست جداشده با کاما (مثلاً `h2,http/1.1`) |
| Fingerprint | اثرانگشت uTLS (مثلاً `chrome`) |
| Security | `inbound_default`، `none`، `tls` یا `reality` |
| Allow insecure | صرف‌نظر از بررسی گواهی |
| Mux | فعال‌سازی مالتی‌پلکسینگ |
| Fragment | مشخصات fragment (مثلاً `1,40-60,30-50`) |
| Priority | ترتیب صعودی درون یک inbound |
| Enabled | روشن/خاموش کردن میزبان |

### متغیرهای قالب

فیلدهای متنی به‌ازای هر کاربر رندر می‌شوند، بنابراین یک تعریف میزبان برای همه کار می‌کند:

| متغیر | به چه چیزی بسط می‌یابد |
|----------|-----------------------|
| `{USERNAME}` | نام کاربری کاربر |
| `{SERVER_IP}` | آدرس نود |
| `{SERVER_PORT}` | پورت تبلیغ‌شده |
| `{PROTOCOL}` | پروتکل inbound (vless، vmess، …) |
| `{NETWORK}` | ترنسپورت (tcp، ws، grpc، …) |
| `{SECURITY}` | امنیت ترنسپورت (tls، reality، none) |
| `{REMARK}` | remark پیکربندی‌شده |

### پیکربندی

1. یک inbound را باز کنید و به تب **Hosts** بروید (یا به **میزبان‌های اشتراک** بروید)
2. روی **Add Host** کلیک کنید
3. فیلدهای بازنویسی موردنیاز را پر کنید — بقیه را خالی بگذارید تا از inbound ارث‌بری شود
4. ردیف‌ها را بکشید (یا از **Reorder** استفاده کنید) تا اولویت تنظیم شود — اولویت کمتر زودتر رندر می‌شود
5. ذخیره کنید. واکشی بعدی اشتراک بلافاصله میزبان‌های جدید را منعکس می‌کند.

!!! tip
    وقتی فقط می‌خواهید آدرس/SNI را تغییر دهید اما همان چیزی که inbound مذاکره می‌کند را نگه دارید، **Security** را روی `inbound_default` بگذارید.

## فرمت‌های جدید خروجی اشتراک

**مکان:** کاربران → جزئیات کاربر → **Subscription** (لینک‌های هر فرمت) و نقطه‌پایانی عمومی `/sub/{token}`

علاوه بر خروجی‌های موجود `base64`، `clash` و `singbox`، نسخه 1.2.3 سه فرمت جدید اضافه می‌کند که با پارامتر کوئری `?format=` قابل انتخاب‌اند:

| فرمت | `?format=` | خروجی |
|--------|-----------|--------|
| Xray / V2Ray JSON | `xray` | JSON خام کلاینت Xray/V2Ray |
| Outline | `outline` | لینک‌های `ss://` شدوساکس برای Outline |
| لینک‌های ساده | `links` | لینک‌های اشتراک به سبک V2rayN، هر کدام در یک خط |

مجموعه‌ی کامل اکنون این است: `base64`، `clash`، `singbox`، `xray`، `outline`، `links`. وقتی فرمتی داده نشود، پاسخ همچنان به‌صورت خودکار از روی User-Agent کلاینت تشخیص داده می‌شود.

```text
https://panel.example.com/sub/<token>?format=xray
https://panel.example.com/sub/<token>?format=outline
https://panel.example.com/sub/<token>?format=links
```

## بسته‌های قانون مسیریابی هوشمند

**مکان:** شبکه و نودها → بسته‌های مسیریابی (Routing Packs)

یک **بسته‌ی مسیریابی** مجموعه‌ای نام‌دار و قابل‌استفاده‌ی مجدد از قوانین مسیریابی است. یک بار بسازید، سپس روی هر نودی اعمال کنید یا در اشتراک Clash/sing-box تعبیه کنید — دیگر لازم نیست همان قوانین مسدودسازی تبلیغات / مستقیم ایران / استریم را به‌ازای هر نود از نو بسازید.

### آنچه می‌توانید انجام دهید

| Action | توضیحات |
|--------|----------|
| ایجاد / ویرایش | ساخت یک بسته از قوانین مسیریابی مرتب (همان فیلدهای مسیریابی نود) |
| اعمال روی نود | جایگزینی قوانین مسیریابی یک نود با بسته و همگام‌سازی مجدد هسته‌ی آن |
| تنظیم پیش‌فرض سراسری | یک بسته در سطح کل ناوگان اعمال می‌شود مگر اینکه بازنویسی شود |
| تخصیص به‌ازای کاربر | به یک کاربر خاص بسته‌ی اختصاصی او را بدهید (در اشتراکش تعبیه می‌شود) |

### پیکربندی

1. روی **New Pack** کلیک کنید و نامی برایش بگذارید (مثلاً «ایران — مسدودسازی تبلیغات + مستقیم»)
2. قوانین را به ترتیب اولویت اضافه کنید (دامنه‌ها، IPها، پورت‌ها، تگ‌های inbound ← outbound/balancer)
3. ذخیره کنید، سپس:
   - بسته را روی یک نود **Apply** کنید تا زنده شود، یا
   - آن را برای کل ناوگان **Default** علامت بزنید، یا
   - از مسیر **کاربران ← کاربر ← Routing Pack** آن را به کاربر تخصیص دهید

!!! note
    تخصیص به‌ازای کاربر بر پیش‌فرض سراسری اولویت دارد. کاربری که تخصیصی ندارد به بسته‌ی پیش‌فرض بازمی‌گردد.

## اسکنر IP تمیز (Clean-IP)

**مکان:** شبکه و نودها → Clean IP

IPهای لبه‌ی Cloudflare/CDN با عملکرد خوب را با اسکن کاندیداها و امتیازدهی بر اساس تأخیر و اتلاف بسته پیدا کنید. بهترین‌ها را به‌عنوان آدرس تبلیغ‌شده در یک میزبان اشتراک یا زنجیره‌ی CDN استفاده کنید.

### نتایج

| Field | توضیحات |
|-------|----------|
| IP | آدرس کاندید |
| Latency (ms) | تأخیر رفت‌وبرگشت |
| Loss % | اتلاف بسته‌ی اندازه‌گیری‌شده |
| Score | رتبه‌بندی ترکیبی (بالاتر بهتر است) |
| Reachable | اینکه آیا پروب متصل شد |
| Scanned at | زمان اندازه‌گیری |

### پیکربندی

1. IPهای کاندید را در کادر اسکن بچسبانید (هر کدام در یک خط)
2. در صورت تمایل یک **port** تنظیم کنید (پیش‌فرض `443`)
3. روی **Scan** کلیک کنید — نتایج امتیازدهی و از بهترین به بدترین مرتب می‌شوند
4. IPهای برتر را در آدرس یک میزبان اشتراک یا یک زنجیره‌ی CDN/Relay کپی کنید

!!! warning "محافظت SSRF"
    اهداف اسکن پیش از پروب اعتبارسنجی می‌شوند. محدوده‌های خصوصی، loopback و link-local رد می‌شوند تا اسکنر نتواند برای دسترسی به سرویس‌های داخلی استفاده شود.

## اعمال محدودیت IP

**مکان:** امنیت → IP Limit

سقف هم‌زمان IP/دستگاه به‌ازای هر کاربر را برای مهار اشتراک‌گذاری حساب اعمال کنید. وقتی کاربری از حد خود عبور می‌کند، اقدام پیکربندی‌شده اجرا می‌شود.

### سیاست

| Setting | توضیحات |
|---------|----------|
| Enabled | روشن/خاموش کردن اعمال |
| Action | `warn`، `disable_temporarily` یا `kill_connections` |
| Alert cooldown | ثانیه‌های بین هشدارهای تکراری برای یک کاربر |
| Restore after | ثانیه‌های پیش از بازگرداندن کاربری که موقتاً غیرفعال شده |

### اقدامات

| Action | رفتار |
|--------|-------|
| **warn** | ثبت یک رویداد و هشدار؛ بدون اقدام بر اتصال |
| **disable_temporarily** | غیرفعال‌سازی کاربر به‌مدت `restore_after` ثانیه، سپس بازگرداندن |
| **kill_connections** | قطع فوری اتصال‌های متخلف |

!!! note "تفاوت‌های هسته"
    `kill_connections` فقط برای **Xray** است. روی نودهای sing-box به‌صورت خودکار به `disable_temporarily` تنزل می‌یابد، چون sing-box نمی‌تواند اتصال‌های زنده‌ی منفرد را قطع کند.

### رویدادها

جدول **Events** هر اقدام اعمال را فهرست می‌کند: کاربر، تعداد IP مشاهده‌شده، خود IPها، اقدام انجام‌شده و زمان.

## پروتکل‌ها و ترنسپورت‌های جدید و ماتریس قابلیت

**مکان:** Inboundها → افزودن/ویرایش (انتخابگرهای پروتکل، ترنسپورت و امنیت)

نسخه 1.2.3 پوشش پروتکل و ترنسپورت را گسترش می‌دهد، و ویرایشگر inbound اکنون انتخاب‌های شما را به آنچه هسته‌ی نودِ انتخاب‌شده واقعاً پشتیبانی می‌کند محدود می‌سازد.

### Xray-core

- **پروتکل‌های ورودی:** `vless`، `vmess`، `trojan`، `shadowsocks` (+ SS-2022 چندکاربره)، `socks`، `http`، `dokodemo`
- **ترنسپورت‌ها:** `tcp`، `ws`، `grpc`، `httpupgrade`، `http`/`h2`، `xhttp`، `mkcp` (mKCP)
- **امنیت:** `none`، `tls`، `reality` — TLS یک فهرست `alpn` می‌گیرد؛ TCP از هدر `none`/`http` پشتیبانی می‌کند؛ xHTTP یک انتخابگر `mode` دارد

### sing-box

- **پروتکل‌های ورودی:** `vless`، `vmess`، `trojan`، `shadowsocks`، `hysteria2`، `tuic`، `wireguard`، `hysteria` (v1)، `shadowtls`، `anytls`، `socks`، `http`، `naive`
- **ترنسپورت‌ها:** `tcp`، `ws`، `grpc`، `httpupgrade`، `http`/`h2`، `quic`

### ماتریس قابلیت به‌ازای پروتکل

پنل یک **ماتریس قابلیت زنده به‌ازای پروتکل** ارائه می‌دهد (`GET /api/capabilities`) و این ماتریس **تنها منبع حقیقت** است — ویرایشگر inbound فقط ترکیب‌های پروتکل/ترنسپورت/امنیتی را پیشنهاد می‌دهد که هسته‌ی انتخاب‌شده پشتیبانی می‌کند.

چند پروتکل **هیچ ترنسپورت استریمی** ندارند:

| پروتکل | هسته | ترنسپورت | امنیت |
|----------|------|-----------|-------|
| `socks` | هر دو | ندارد (TCP خام) | plaintext |
| `http` | هر دو | ندارد (TCP خام) | plaintext |
| `naive` | sing-box | ندارد | **TLS اجباری** |
| `dokodemo` | xray | ندارد (TCP/UDP خام) | plaintext |

!!! warning
    inboundهای `socks` و `http` به‌صورت **plaintext** هستند — فقط روی شبکه‌های مورد اعتماد یا پشت یک relay محلی آن‌ها را در معرض قرار دهید. `naive` به‌صورت **اجباری TLS** را الزام می‌کند.

برای نمونه‌های کامل پیکربندی، [پروتکل‌ها](13-protocols-config.md) و `docs/protocols.md` را ببینید.
